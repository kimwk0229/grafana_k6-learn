# 부하 테스트

업계에서 흔히 저지르는 실수는 *성능 테스트*와 *부하 테스트*라는 용어를 혼용하는 것입니다. 이 둘은 같지 않습니다! 오히려 부하 테스트는 성능 테스트의 하위 실천법입니다.

**성능 테스트 != 부하 테스트**

[성능 테스트](01-Introduction-to-Performance-Testing.md)는 확장성, 탄력성, 가용성, 신뢰성, 복원성, 지연 시간 등의 측면을 포함하여 시스템이 전반적으로 얼마나 잘 작동하는지 검증합니다. 부하 테스트는 성능 테스트의 한 유형으로, 애플리케이션 성능의 많은 측면을 테스트하는 데 사용할 수 있는 접근 방식입니다. 그러나 모든 성능 테스트가 부하 테스트를 포함하는 것은 아닙니다.

부하 테스트는 처리해야 할 작업의 양이 상당할 때 애플리케이션의 성능을 검증하고 확인하는 데 특히 초점을 맞춥니다. 부하 테스트는 수동으로 수행할 수 있지만, 대부분의 팀은 애플리케이션에 접근하는 실제 사용자를 프로그래밍 방식으로 시뮬레이션하기 위해 자동화된 부하 테스트 스크립트를 작성합니다.

## 테스트 파라미터

부하 테스트 스크립트를 작성할 때는 생성할 부하의 양 외에도 많은 요소를 고려해야 합니다. _테스트 파라미터_라고 불리는 이러한 요소들에는 부하의 분포, 형태, 패턴이 포함됩니다.

다음은 일반적인 테스트 파라미터의 일부 목록입니다:

- **가상 사용자(VU, Virtual Users).** VU는 다른 VU 스레드와 동시에 실행되는 독립적인 실행 스레드입니다. 보통 스크립트는 1명의 VU 활동이 1명의 실제 사용자를 나타내도록 설계됩니다.
- **반복(Iterations)**. VU가 실행할 스크립트의 총 반복 횟수입니다.
- **처리량(Throughput)**. 테스트가 시간에 따라 생성하는 부하의 양을 측정한 것으로, 보통 초당 VU, 초당 요청 수, 또는 초당 반복 수로 정의됩니다.
- **사용자 흐름(User flows)**. 스크립트가 수행하는 작업과 그 순서입니다. 일반적으로 사용자 흐름은 애플리케이션을 통한 실제 사용자의 경로를 나타냅니다.
- **부하 프로필(Load profile)**. 시간 경과에 따라 테스트가 생성하는 트래픽의 형태입니다. 지연 시간(think time)의 횟수, 테스트가 시간에 따라 VU 수를 점진적으로 증가 또는 감소시키는 램프업 및 램프다운 기간, 그리고 테스트의 단계(stages)가 포함됩니다.
- **지속 시간(Duration)**. 전체 테스트와 각 단계를 실행하는 데 걸리는 시간입니다.

## 부하를 시뮬레이션하는 방법

일반적으로 부하를 시뮬레이션하는 몇 가지 방법이 있습니다:

**프로토콜 기반 부하 테스트**는 애플리케이션 서버로 가는 기반 요청을 시뮬레이션합니다. 이러한 요청은 프로토콜 레이어를 통해 전송됩니다. 프로토콜 기반 부하 테스트는 범위가 다양하며, 다음을 모두 포함합니다:
- 하나 이상의 특정 애플리케이션 구성 요소에 대한 타겟 API 테스트
- 전체 시스템을 통해 여러 구성 요소를 거치는 트래픽을 시뮬레이션하는 엔드투엔드 또는 더 현실적인 테스트

**브라우저 기반 부하 테스트**는 반면에 사용자가 애플리케이션 사용자 인터페이스와 상호작용하는 방식을 시뮬레이션합니다. 프로토콜 수준의 요청을 시뮬레이션하는 대신, 브라우저 기반 테스트는 웹 앱의 요소 클릭, 양식 입력, 실제 사용자가 취할 수 있는 기타 작업 등을 자동화합니다. 브라우저 기반 부하 테스트는 엔드투엔드 테스트를 더 자주 포함합니다.

**하이브리드 부하 테스트**는 프로토콜 기반 테스트와 브라우저 기반 테스트의 혼합입니다. 이는 보통 성능의 다양한 측면을 테스트하기 위해 사용됩니다. 가장 경제적인 접근 방식은 프로토콜 기반 부하 테스트로 대부분의 부하를 생성하고, 더 적은 수의 브라우저 기반 부하 테스트 사용자를 함께 사용하는 것입니다.

## 부하 테스트 시나리오

부하 테스트 시나리오는 특정 테스트 파라미터 값의 조합입니다. 각 시나리오는 애플리케이션이 노출될 특정 상황이나 조건 집합을 재현합니다.

부하 테스트 시나리오는 종종 _부하 테스트 유형_이라고 불립니다. 가장 일반적인 시나리오들이 여기에 나열되어 있습니다.

### 쉐이크아웃 테스트(Shakeout Test)

쉐이크아웃 테스트는 스모크 테스트라고도 하며, 더 많은 시간과 리소스를 사용하기 전에 주요 문제를 확인하는 소규모 테스트입니다. 쉐이크아웃 테스트는 일반적으로 짧은 시간 동안 하나 또는 몇 개의 VU를 사용하며 다음과 같은 주요 문제를 확인합니다:
- 테스트 결과의 정확성에 크게 영향을 미치는 스크립트 관련 문제
- 예상치 못한 환경 구성
- 낮은 부하에서도 발생하는 상당한 애플리케이션 성능 병목 현상

쉐이크아웃 테스트가 실패하면, 발견된 문제를 먼저 해결한 후에 테스트 처리량을 늘려야 합니다.

_예시:_
![](../../images/scenarios-shakeout.png)

```
Name: Shakeout Test
Total VUs: 5	
Ramp-up: 0 seconds	
Duration: 10 minutes
Ramp-Down: 0 seconds
```


쉐이크아웃 테스트는 [여러 시나리오](../III-k6-Intermediate/09-Workload-modeling-with-scenarios.md)로 구성될 수도 있습니다.

### 평균 부하 테스트(Average Load Test)

이 시나리오는 프로덕션의 일반적인 한 시간 동안의 사용자 워크로드를 시뮬레이션합니다. 이 시나리오는 해당 시간 내에서 가장 자주 실행되는 요청 또는 기능을 포함합니다.

이 테스트 시나리오는 일반적으로 사용자가 점진적으로 로그인하고 시스템과 상호작용하는 것을 시뮬레이션하기 위한 램프업 및 램프다운 기간을 포함합니다. 부하 테스트는 프로덕션의 평균 부하 동작을 모방하기 위해 원하는 부하에 도달할 때까지 VU 수를 점진적으로 증가시킵니다.

램프업과 램프다운 기간 사이에는 가상 사용자 수가 일정하게 유지되는 정상 상태(steady state) 기간이 있습니다.
평균 부하 테스트에서는 약 1시간 동안 정상 상태 부하 시뮬레이션을 유지합니다.

_예시_:

![](../../images/test-scenario-average.png)
```	
Name: Average Load Test
Total VUs: 100	
Ramp-up: 30 minutes
Steady state: 60 minutes
Ramp-down: 10 minutes
Total duration: 100 minutes
```

앞의 예시에서 평균 부하 테스트는 VU 수로 정의되었지만, 테스트가 생성하는 반복 횟수나 초당 요청 수로도 정의할 수 있습니다. 더 많은 정보는 [executor를 이용한 부하 프로필 설정](../III-k6-Intermediate/08-Setting-load-profiles-with-executors/Setting-load-profiles-with-executors.md)을 참고하세요.

### 스트레스 테스트(Stress Test)

스트레스 테스트는 최대 부하 테스트(peak load test)라고도 하며, 하루 또는 시즌 중 *가장 높은* 시점에 애플리케이션이 경험할 것으로 예상되는 트래픽을 시뮬레이션합니다. 평균 부하 테스트가 주, 월 또는 그 이상에 걸쳐 집계된 일반적인 날의 트래픽을 시뮬레이션하는 반면, 스트레스 테스트는 애플리케이션이 경험하는 가장 높은 양의 트래픽에 초점을 맞춥니다.

스트레스 테스트는 형태상 평균 부하 테스트와 유사하지만, 일반적으로 훨씬 높은 테스트 처리량을 생성합니다. "정상" 시간대에는 100명의 사용자가 접근하지만 점심 시간에는 300명의 사용자가 접근하는 애플리케이션을 고려해보세요. 이 애플리케이션은 300 VU 수준의 스트레스 테스트로 테스트하면 도움이 될 수 있습니다.

스트레스 테스트는 애플리케이션이 비정상적으로 높은 부하에 직면하는 러시아워나 세일 기간을 테스트할 때 좋은 테스트 시나리오입니다.

### 소크 테스트 또는 지구력 테스트(Soak or Endurance Test)

소크 테스트는 지구력 테스트라고도 하며, 평균 또는 최대 테스트보다 더 긴 기간의 테스트입니다. 메모리 관리 결함으로 인한 것과 같은 일부 성능 병목 현상은 더 긴 기간이 지나야만 나타납니다. 소크 테스트는 시간이 지남에 따라 성능이 저하되는지 여부를 확인합니다.

이 시나리오는 평균 부하 테스트와 유사한 테스트 처리량 수준을 갖지만, 애플리케이션에 따라 여러 시간 또는 며칠까지 연장됩니다.
  
_예시_:

![](../../images/test-scenario-soak.png)

```		
Name: Soak Test
Total VUs: 50
Ramp-up: 30 minutes
Steady state: 480 minutes
Ramp-down: 10 minutes
Total duration: 520 minutes (8 hours and 40 minutes)
```

### 스파이크 테스트(Spike Test)

이전의 모든 테스트 유형은 부하가 점진적으로 도입되거나(평균, 스트레스, 소크 테스트의 경우) 소량의 부하가 실행되는(쉐이크아웃) 상황을 재현했습니다. 반면 스파이크 테스트는 애플리케이션이 *갑작스럽고* 대규모적인 트래픽 증가를 경험하는 상황을 재현합니다.

스파이크 테스트는 트래픽이 짧은 시간에 낮은 수준에서 극도로 높은 수준으로 올라가는 시간대의 애플리케이션 성능을 검증하는 데 사용할 수 있습니다. 스파이크 테스트는 다음과 같은 특정 시간에 예정된 이벤트를 시뮬레이션하는 데 유용합니다:
- 주요 제품 발표(슈퍼볼 광고와 같은)
- 제품 출시 또는 콘서트 티켓 판매
- 마감일(세금 신고 마지막 날)
- 세일 시즌 개막(블랙 프라이데이 또는 사이버 먼데이)

스파이크 테스트는 높은 처리량과 짧은 정상 상태 기간을 갖습니다. 일반적으로 램프업 및 램프다운 기간이 무시할 수 있을 정도로 짧습니다.

이전 시나리오와의 또 다른 차이점은 테스트할 사용자 흐름의 선택입니다. 일상적인 다양한 프로세스 대신, 스파이크 테스트는 보통 하나의 사용자 흐름을 우선시합니다.
예를 들어, 이벤트 티켓 판매에서 사용자는 다른 페이지를 탐색하는 것보다 티켓 구매에 집중할 수 있습니다.

_예시_:

![](../../images/test-scenario-spike-test.png)
```
Name: Spike Test	
Total VUs: 300
Ramp-up: 1 minute
Steady state: 20 minutes
Ramp-down: 5 minutes
Total duration: 26 minutes
```



### 브레이크포인트 테스트(Breakpoint Test)

이전의 부하 테스트 유형들이 현실적이고 예상되는 프로덕션 부하를 시뮬레이션하는 반면, 브레이크포인트 테스트는 한 단계 더 나아가려 합니다. 브레이크포인트 테스트는 성능이 저하되기 시작하는 트래픽 수준을 파악하기 위해 애플리케이션에 점점 더 높은 수준의 부하를 가합니다.

브레이크포인트 테스트는 시스템이 처리할 수 있는 것에 대한 신뢰를 구축합니다. 브레이크포인트 테스트 결과는 용량 계획을 위한 귀중한 입력을 제공합니다.

브레이크포인트 테스트 시나리오는 다른 시나리오보다 램프업 기간에 더 집중합니다. 완전히 점진적인 램프업으로만 구성될 수도 있고, 램프업 기간과 정상 상태 기간이 번갈아 나타나는 형태로 구성될 수도 있습니다. 다음 예시의 단계적 부하 프로필 패턴은 성능 저하와 특정 부하 수준을 상관시키는 데 도움이 됩니다.

_예시_:

![](../../images/test-scenarios-breakpoint.png)
```
Name: Breakpoint Test	
Total max VUs: unknown
Ramp-up: 10 minutes before each stage
Steady state: 30 minutes
Ramp-down: 0 minutes
Total duration: unknown
```


브레이크포인트 테스트는 본질적으로 더 탐색적이기 때문에, 테스터는 사전에 최대 VU 수나 테스트 기간을 알 수 없습니다. 팀은 일반적으로 브레이크포인트 테스트가 실행되는 동안 애플리케이션을 면밀히 모니터링하다가, 특정 [임계값(thresholds)](../II-k6-Foundations/07-Setting-test-criteria-with-thresholds.md)을 초과하면 테스트를 수동으로 중단하거나 자동으로 중단되도록 프로그래밍합니다.

## 지식 확인

### 문제 1

다음 중 성능 모범 사례에 대한 진술로 참인 것은?

A: 선택된 부하 테스트 시나리오는 테스트 결과의 정확성에 크게 영향을 미칠 수 있다.

B: 성능 테스트를 수행하는 가장 좋은 방법은 프로덕션에서 애플리케이션에 접근할 사용자의 최대 부하를 시뮬레이션하는 것이다.

C: 테스트에서 가상 사용자 수는 시나리오에서 가장 중요한 테스트 파라미터다.

### 문제 2

새로운 애플리케이션과 환경에서 첫 번째 테스트 실행에 사용해야 할 테스트 시나리오는?

A: 소크 테스트

B: 쉐이크아웃 테스트

C: 회귀 테스트

### 문제 3

메모리 누수 테스트에 가장 적합한 테스트 시나리오는?

A: 스트레스 테스트

B: 소크 테스트

C: 평균 부하 테스트

### 정답

1. A. B는 최대 부하 테스트라는 한 가지 유형의 부하 테스트를 설명하며, C는 선택한 테스트 시나리오와 테스트 결과에 영향을 미칠 수 있는 많은 테스트 파라미터 중 하나만 언급합니다.
2. B. 소크 테스트는 더 긴 기간 동안 실행되며, 환경이 쉐이크아웃 테스트를 거치지 않은 경우 너무 많은 오류로 종료되어 유용하지 않을 수 있습니다. 전체 회귀 테스트도 환경과 스크립트가 작동하는지 먼저 확인하지 않으면 유용하지 않을 수 있습니다.
3. B. 메모리 누수는 스트레스 테스트와 평균 부하 테스트 중에도 발견될 수 있지만, 소크 테스트의 긴 기간은 메모리 누수와 같은 더 점진적인 추세를 발견하는 데 독특하게 적합합니다.
